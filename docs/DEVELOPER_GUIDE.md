# Joomla MCP Server: Complete Developer Guide

## What This Is

This is a **Model Context Protocol (MCP) server** that lets an AI assistant (Claude) manage a Joomla 3 admin backend by automating the HTML admin interface — logging in, extracting CSRF tokens, filling forms, and submitting them. No REST API is involved; it works exactly like a browser automation tool.

**Why form-scraping?** Joomla 3 has no usable REST API. Every operation GETs the admin form first (to get a fresh CSRF token), then POSTs with all form fields preserved, then verifies the result by re-fetching.

---

## Architecture

```
Claude (AI) → MCP Protocol → src/index.ts → JoomlaClient → Joomla Admin HTML
```

**Two main files:**

| File | Role |
|---|---|
| `src/index.ts` | MCP server: ~62 tool definitions, input validation, delegates to JoomlaClient |
| `src/joomla-client.ts` | All Joomla logic: session, cookies, CSRF tokens, HTML parsing, form submission |

**Supporting directories:**

| Directory | Purpose |
|---|---|
| `blueprints/` | YAML/JSON snapshots of modules/outlines for cross-site reuse |
| `snapshots/` | Point-in-time form captures for undo/restore operations |
| `scripts/tests/` | Integration tests that hit a real Joomla site |
| `docs/` | Roadmap and planning docs |
| `dist/` | Compiled JS output (gitignored, regenerated by `npm run build`) |

## How to Run It

### 1. Configure credentials

Copy `.env.example` to `.env` and fill in:

```env
JOOMLA_BASE_URL=https://yoursite.com/
JOOMLA_USERNAME=your-admin-user
JOOMLA_PASSWORD=your-admin-password
```

The base URL can be `https://yoursite.com/` or `https://yoursite.com/administrator` — the code normalizes both.

### 2. Install dependencies

```powershell
npm install
```

### 3. Run in development (with Claude Desktop or another MCP client)

**Option A: Run directly via tsx (no compile step)**
```powershell
npm start          # tsx src/index.ts
npm run dev        # tsx watch src/index.ts (auto-reloads on save)
```

**Option B: Compile then run**
```powershell
npm run build      # tsc → dist/
node dist/index.js
```

### 4. Connect Claude Desktop to this server

In Claude Desktop's `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "joomla": {
      "command": "node",
      "args": ["C:\\path\\to\\joomla-mcp-master\\dist\\index.js"],
      "env": {
        "JOOMLA_BASE_URL": "https://yoursite.com/",
        "JOOMLA_USERNAME": "admin",
        "JOOMLA_PASSWORD": "secret"
      }
    }
  }
}
```

Or point to `tsx` for development:
```json
{
  "command": "npx",
  "args": ["tsx", "C:\\path\\to\\joomla-mcp-master\\src\\index.ts"]
}
```

### 5. Use the MCP Inspector for debugging

```powershell
npm run inspect    # opens browser UI to call tools manually
```

This is invaluable during development — you can call any tool with custom arguments and see raw responses without involving Claude.

## How the Code Works (Mental Model)

### Request lifecycle for any tool call

1. **Claude calls a tool** (e.g., `joomla_update_article` with `{id: "42", title: "New Title"}`)
2. **`index.ts` handler** validates required params, calls `joomla.updateArticle({...})`
3. **`JoomlaClient` method** does the GET-POST-verify dance:
   - `getPage(articleEditUrl)` → gets HTML, extracts CSRF token, stores cookies
   - Extracts all existing form fields from HTML (preserves hidden fields Joomla requires)
   - Applies only the changed fields on top
   - `postPage(articleEditUrl, formData)` → POSTs with CSRF token, follows redirect
   - Re-fetches the form to verify the change took effect
4. **Returns `JoomlaResponse`**: `{ success: boolean, message: string, data?: unknown }`
5. **`index.ts` wraps it** in MCP format and returns to Claude

### Session & cookie management

```typescript
private cookies: Map<string, string> = new Map();
```

Every `request()` call reads from `this.cookies` and writes back any `Set-Cookie` headers it receives. The session is maintained in memory for the lifetime of the MCP server process. If you restart the server, it re-logs in on first use.

### CSRF token extraction

Joomla requires a CSRF token on every POST. The code tries three fallback methods:
1. Extract from JS options JSON: `"csrf.token": "value"`
2. Extract from hidden input with CSRF markers
3. Extract from any hidden input with a 32-char hex name

The token is cached as `this.tokenName` and refreshed on every `getPage()` call.

### Safety features built in

- **Expected title checks**: Tools accept `expectedTitle` — if the fetched entity doesn't match, the operation is refused. Prevents "I had the wrong ID" accidents.
- **Dry-run defaults**: Destructive operations (delete, restore snapshot, site build apply) default to `dryRun: true`. You must pass `confirm: true` to execute.
- **Snapshot/restore**: Before risky operations, capture state with `joomla_snapshot_target`. Restore with `joomla_restore_snapshot`.
- **Check-in**: Joomla locks records when editing. If something crashes mid-edit, use `joomla_checkin_*` tools to unlock.

## How to Develop a New Tool

Follow this exact pattern — don't deviate from it:

### Step 1: Add the method to `JoomlaClient` (`src/joomla-client.ts`)

```typescript
async myNewOperation(params: { id: string; newValue: string }): Promise<JoomlaResponse> {
  // 1. Build the URL
  const url = `${this.getAdminUrl()}?option=com_something&task=thing.edit&id=${params.id}`;
  
  // 2. GET the form to get current state + CSRF token
  const { html } = await this.getPage(url);
  
  // 3. Extract all existing form fields (CRITICAL — Joomla rejects partial POSTs)
  const fields = this.extractFormFields(html);
  
  // 4. Overlay only your changes
  fields["jform[myfield]"] = params.newValue;
  fields["task"] = "thing.apply";
  
  // 5. POST
  const result = await this.postPage(url, fields);
  
  // 6. Verify by re-fetching
  const { html: verifyHtml } = await this.getPage(url);
  const verified = verifyHtml.includes(params.newValue);
  
  return {
    success: verified,
    message: verified ? "Updated successfully" : "Update may have failed",
    data: { id: params.id },
  };
}
```

**Key rule:** Always call `extractFormFields()` first and overlay your changes on top. Never construct a POST body from scratch — Joomla forms have many required hidden fields.

### Step 2: Register the tool in `index.ts` (`src/index.ts`)

In the `ListToolsRequestSchema` handler (the big array of tool definitions):

```typescript
{
  name: "joomla_my_new_operation",
  description: "Does the thing",
  inputSchema: {
    type: "object",
    properties: {
      id: { type: "string", description: "The entity ID" },
      newValue: { type: "string", description: "The new value to set" },
    },
    required: ["id", "newValue"],
  },
},
```

In the `CallToolRequestSchema` handler (the big switch statement):

```typescript
case "joomla_my_new_operation": {
  const login = await ensureLoggedIn();
  if (!login.success) return { content: [{ type: "text", text: formatResult(login) }], isError: true };

  const id = args?.id as string;
  const newValue = args?.newValue as string;
  if (!id || !newValue)
    return { content: [{ type: "text", text: "Error: id and newValue are required" }], isError: true };

  const result = await joomla.myNewOperation({ id, newValue });
  return {
    content: [{ type: "text", text: formatResult(result) }],
    isError: !result.success,
  };
}
```

### Step 3: Build and test

```powershell
npm run build           # check for TypeScript errors
npm run inspect         # call your tool manually in the browser UI
```

## How to Fix Bugs

### Debugging workflow

**1. Reproduce with MCP Inspector first**
```powershell
npm run inspect
```
Call the failing tool with the exact args. The inspector shows the raw JSON response including the `htmlPreview` field (first 2000 chars of the page HTML). This often tells you exactly what Joomla returned.

**2. Add temporary logging to `joomla-client.ts`**

The `request()` method is the lowest level. Add `console.error()` calls (not `console.log` — stdout is the MCP protocol channel and will corrupt it):
```typescript
console.error("DEBUG URL:", url);
console.error("DEBUG status:", response.status);
console.error("DEBUG html snippet:", html.substring(0, 500));
```

**3. Use `joomla_page_content`**

This tool returns raw HTML for any admin URL. Use it to inspect what Joomla is actually rendering:
```
joomla_page_content({ path: "?option=com_content&task=article.edit&id=42" })
```

**4. Use `joomla_inspect_admin_form`**

This tool extracts and lists all form fields for a given admin page — essential for knowing what fields exist:
```
joomla_inspect_admin_form({ path: "?option=com_content&task=article.edit&id=42" })
```

### Common failure patterns

| Symptom | Likely Cause | Fix |
|---|---|---|
| "Login failed" | Wrong credentials or site URL | Check `.env`, verify you can log in manually |
| Operation appears to succeed but nothing changed | CSRF token stale or missing | Check that `getPage` was called before `postPage` |
| "Record not found" after create | Joomla returned an error page | Check the `htmlPreview` in the response for Joomla's error message |
| Checkin needed | Item was left in editing state | Call `joomla_checkin_*` tool for that item |
| Gantry operations fail | Stale Gantry URL cache | Re-login with `joomla_login` to clear caches |
| Form POST rejected | Missing required hidden fields | Ensure `extractFormFields()` was called first |
| TypeScript compile error | Strict null checks | Never use `!` assertions — check for null/undefined explicitly |

## How to Maintain It

### Keeping Joomla changes from breaking things

Joomla admin HTML can change across minor versions. When an upgrade breaks a tool:

1. Use `joomla_page_content` to see the new HTML
2. Use `joomla_inspect_admin_form` to see new field names
3. Update the field extraction logic in `joomla-client.ts`
4. Run `npm run smoke` to verify connectivity, then `npm run test:all-tools` for full verification

### When to use snapshots

Before any risky or complex operation:
```
joomla_snapshot_target({ kind: "article", id: "42" })
# → returns snapshotId like "snapshot-2025-05-06-article-42-abc123"
```

Store that ID. If something goes wrong:
```
joomla_restore_snapshot({ snapshotId: "...", confirm: true })
```

### Blueprint system

Export a module's configuration as reusable YAML:
```
joomla_export_module_blueprint({ id: "15", format: "yaml", saveToFile: true })
```

This saves to `blueprints/`. Import it on another site or after a rebuild:
```
joomla_import_module_blueprint({ filePath: "blueprints/mymodule.yaml", confirm: true })
```

## How to Troubleshoot It

### Is the server running?

MCP clients connect via stdio (stdin/stdout pipe). The server process is spawned fresh by the client each time. If tools aren't available in Claude:
1. Check `claude_desktop_config.json` — is the path correct?
2. Run the command manually in terminal — does it start without errors?
3. Check Claude Desktop's MCP server logs (usually in `%APPDATA%\Claude\logs\`)

### Is authentication working?

```
joomla_login  (no args needed)
```

If it fails, double-check `.env` values and whether the Joomla site is accessible from your machine.

### Is it hitting the right Joomla instance?

```
joomla_page_content({ path: "" })  # or path: "/"
```

Check `htmlPreview` — is it the right site?

### Why did an operation fail silently?

Every response includes:
- `success: true/false`
- `message: "..."` — human-readable status
- `htmlPreview` — first 2000 chars of the page Joomla returned

The `htmlPreview` is your best debugging tool. Joomla often puts error messages in the page body rather than returning HTTP error codes.

### Gantry-specific troubleshooting

Gantry caches layout data internally. If you get "stale snapshot" type errors:
1. Call `joomla_login` to clear all Gantry URL caches
2. Retry the operation

Gantry also uses its own admin routes separate from core Joomla — these are auto-detected and cached in `gantryEntryUrl` on `JoomlaClient`.

## How to Make It Work Well

### Best practices for AI-driven use

1. **Always list before acting**: `joomla_list_articles` before `joomla_update_article` — confirms IDs are correct
2. **Use expectedTitle**: Pass `expectedTitle` on updates and deletes — the server will refuse if the ID doesn't match the expected entity
3. **Snapshot before bulk operations**: Take a snapshot before doing multi-step workflows
4. **Use dry-run first**: For `joomla_apply_site_build`, `joomla_submit_admin_form`, etc. — always run with `dryRun: true` first, review, then `confirm: true`
5. **Check-in locked items**: If Claude crashes mid-edit, use `joomla_checkin_*` to unlock

### Performance considerations

- Each tool call that modifies data does at least 2 HTTP requests (GET form + POST + GET verify)
- Complex operations (site build, Gantry layout save) do many more
- The server is single-threaded; don't try to parallelize tool calls
- Session persists for the lifetime of the process — no need to call `joomla_login` before each operation (it's called automatically on first use via `ensureLoggedIn()`)

### Running tests

```powershell
npm run smoke                      # quick connectivity test
npm run test:all-tools             # full integration suite (hits live Joomla)
npm run test:tools:content         # only article/category tests
npm run test:tools:modules         # only module tests
npm run test:tools:gantry          # only Gantry 5 tests
npm run test:tools:menus           # only menu tests
npm run test:tools:admin           # only admin/introspection tests
```

Tests require a real Joomla site — they're integration tests, not unit tests.

### TypeScript gotchas in this codebase

- `console.log()` **corrupts the MCP protocol** (stdout is the protocol channel). Use `console.error()` for any debugging output.
- The codebase uses `strict: true`. Never bypass TypeScript with `as any` or `!` — fix the types properly.
- `tsx` runs TypeScript directly without compiling. The compiled `dist/` output is only needed for production/MCP client use.
